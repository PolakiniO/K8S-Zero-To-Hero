# Lab - Week 3

**Assignment:** 

# Week 2 Labs: Networking + Deployment Incidents

Objective: Learn to troubleshoot real-world production failures involving networking, rollouts, storage, and application behavior.

These labs simulate the types of incidents that actually happen in clusters.

---

## Lab 6 - Label Selector Mismatch (Service cannot reach Pod)

### Tasks

1. Deploy a pod with a label.
2. Create a Service with the wrong selector.
3. Observe no endpoints.
4. Fix selector and verify connectivity.

### Validation Commands

```bash
kubectl get svc
kubectl get endpoints
kubectl describe svc <service>
kubectl get pod --show-labels
```

### Expected Learning

- Services route only by label selectors
- Endpoints object is the first place to look when traffic fails

### Interview Phrase

Service selector mismatch caused routing failure.

---

## Lab 7 - Rolling Update Failure

### Tasks

1. Create a Deployment with a working image.
2. Rollout a broken image version.
3. Observe rollout failure.
4. Inspect rollout status and pods.
5. Roll back to last working version.

### Validation Commands

```bash
kubectl rollout status deployment <name>
kubectl get pods
kubectl describe pod <pod>
kubectl rollouthistory deployment <name>
kubectl rollout undo deployment <name>
```

### Expected Learning

- How to detect failing rollouts
- How ReplicaSets behave during deployment failures
- Safe rollback procedures

### Interview Phrase

I inspected rollout status and isolated failing replicas.

---

## Lab 8 - Storage Failure (PVC Pending)

### Tasks

1. Create a PersistentVolumeClaim.
2. Use a storage class that does not exist.
3. Observe PVC stuck Pending.
4. Diagnose why binding failed.

### Validation Commands

```bash
kubectl get pvc
kubectl describe pvc
kubectl get storageclass
```

### Expected Learning

- Pod scheduling can be blocked by storage
- PVC Pending is a common production issue

### Interview Phrase

Pod was blocked by volume provisioning.

---

## Lab 9 - Network Policy Failure

### Tasks

1. Deploy two pods.
2. Verify connectivity.
3. Apply a NetworkPolicy blocking traffic.
4. Observe connection failure.
5. Diagnose and allow traffic again.

### Validation Commands

```bash
kubectl get networkpolicy
kubectlexec <pod> -- wget -qO- <service-ip>
kubectl describe networkpolicy
```

### Expected Learning

- NetworkPolicies are deny-by-default once applied
- Debugging east-west traffic

### Interview Phrase

Network policy prevented east-west communication.

---

## Lab 10 - Incident Simulation (Pod Running but Broken)

### Tasks

1. Deploy an app that starts but fails internally.
2. Observe pod is Running but service is broken.
3. Use logs, exec, describe to find root cause.
4. Fix the issue.

### Validation Commands

```bash
kubectl logs <pod>
kubectlexec -it <pod> -- sh
kubectl describe pod <pod>
```

### Expected Learning

- Running does not mean healthy
- Structured debugging workflow

### Interview Phrase

I treated it as an incident investigation.
